# Plan: Simplification of Grid Live Architecture (V20260129_1038)

This plan removes the "Leader Election" complexity and flattens `grid_live.json` into a simple list of explicitly tracked strategies.

## 1. Requirements
- Change `grid_live.json` from a dictionary of groups to a flat list of strategy/product/metric objects.
- Stop the UI from automatically switching "Leaders" based on P&L.
- Simplify the monitor loop to process this flattened list.

## 2. Plan of Approach

### Data Format Change
- `grid_live.json` will now be: `[ { "model": "...", "product": "...", "metric": "...", "activated_at": "..." }, ... ]`

### Grid Monitor (`grid_live_monitor.py`)
- [x] Update `monitor_loop` to handle `live_config` as a list instead of a dictionary.
- [x] Remove the grouping logic and iterate through the list once.

### API Server (`trade_viewer_api.py`)
- [x] Update `get_grid_live` to return the new list format.
- [x] Update `update_grid_live` to accept the new format (or continue accepting group-based but flatten it before saving). Actually, it's better to simplify the API to just "replace the whole list".
- [x] Update `_sync_grid_to_activations` to handle the flat list.

### UI Logic (`LiveMonitor.js` & `multi_chart.js`)
- [x] Refactor `LiveMonitor.js` to remove all leader election code (`_updateLeader`, `currentLeaders`, etc.).
- [x] Change its `update` method to simply report the current set of overlays that have "Live" toggled ON.
- [x] In `multi_chart.js`, the "LIVE" toggle will now just add/remove that specific (strat, prod, metric) tuple to a master "Live Set".
