# Multi-Source Grouped L-Trade Requirements

## Version
- Draft Timestamp: 2026-02-13
- Status: Approved conversation requirements (implementation spec)

## Objective
Enable concurrent L-trade generation from multiple automation sources while isolating risk controls, targets, and monitoring by source-group.

## Core Functional Change
Current behavior: grid/live workflow effectively enforces one dominant source at a time.

Required behavior:
- Allow multiple sources concurrently.
- Preserve origin grouping for every promoted L-trade.
- Treat each source-group as an independent “live book” for limits/targets/monitoring.

## Source-Group Model
### Grouping Key
- Use source-group identity based on `source + group` with fallback to source.
- Examples:
  - `strategy_performance|GBPNZD_C`
  - `TB_0212_1603_EURAUD_C_breakout_R_2_tp10.0_sl50.0|0212_1603_EURAUD_C_breakout_R_2_tp10.0_sl50.0`
  - `frequency|<group>`
  - fallback: `breakout`

### Trade Metadata Requirements
Every trade promoted (or evaluated for promotion) must carry:
- `source_screen` (existing behavior)
- `source_group` (new explicit grouping field)
- `trade_block_reason` (existing)
- `trade_block_reason_detail` (existing enhanced detail)
- `grid_live_context_at_block` (existing captured context)

## Risk Controls (Per Source-Group)
### 1) max_live_trades
- Must apply per source-group, not globally.
- Concurrent trades from different groups are allowed.
- Within one source-group, `max_live_trades` remains strict/absolute.

### 2) daily_target
- Must apply per source-group, not globally.
- `daily_target` value is duplicated per group (same target for each group).
- Future global target may be added later but is out of scope now.

## Grid Behavior (Critical)
### Multi-Source Insert/Update Rules
When multiple sources are enabled:
- Adding an item from a different source-group must append, not replace.
- Same `(model, product, metric)` from different source-groups must coexist as separate rows.
- Update/replace is only allowed within the same source-group identity.
- Entries belonging to other source-groups must never be overwritten by a new source-group write.

## Live Trades Screen Requirements
### Partitioning
Live-trades view must be partitioned by source-group and show each group as an independent set:
- Open count per group
- Closed count per group
- Group P&L (running and closed)
- Group target progress
- Group max-live utilization

### Comparison UX
Must make real-time comparison easy between groups:
- Side-by-side or clearly segmented sections.
- Consistent per-group metrics and ordering.
- Fast visual indication of best/worst performing group.

## Config UI Requirements
### Multi-Source Selection
Current single-select source must be replaced with multi-select capability.

Required config shape:
- Keep backward compatibility with existing `automated_trade_source`.
- Add multi-source list, e.g. `automated_trade_sources`.
- If list is present and non-empty, it is authoritative.
- If list missing, fallback to single source behavior.

### Validation Rules
- Prevent invalid/unknown sources.
- Preserve existing source names (`Top20`, `PSB`, `PSB_Family`, `Trade Bucket`, `Frequency`).
- Allow additional source tags if future modules add them.

## API / Data Contract Requirements
### Grid APIs
- Must return enough metadata to identify source-group per row.
- Must support coexistence of duplicate model/product keys across different source-groups.

### Promotion/Block Diagnostics
- Existing promotion-block diagnostics must remain source-group aware.
- Context payload must include group identity and competing group entries for same model/product if applicable.

## Backward Compatibility
- Existing files with no `source_group` must still work.
- Fallback grouping for legacy records:
  - `source_group = source_screen` if present
  - otherwise `source_group = breakout`

## Out of Scope (for this phase)
- Global cross-group daily target.
- Repricing/rebalancing between groups.
- Cross-group auto-pruning logic.

## Acceptance Criteria
1. Multiple sources can populate grid concurrently without replacing each other.
2. L-trades can be created concurrently from different source-groups.
3. `max_live_trades` blocks within-group overflow only.
4. `daily_target` blocks new trades only for the group that hit its target.
5. Live-trades screen clearly partitions and compares groups.
6. Config UI allows selecting multiple sources.
7. Block reasons and diagnostics explicitly include source-group context.

## Test Scenarios
1. Two sources, same model/product/metric
- Expected: two grid rows (different source-group), both preserved.

2. Group A at cap, Group B below cap
- Expected: Group A new promotions blocked; Group B promotions allowed.

3. Group A hits daily target, Group B not hit
- Expected: Group A new promotions blocked; Group B continues.

4. Legacy trade file without `source_group`
- Expected: grouped under fallback source, no crash.

5. Promotion-block diagnostics
- Expected: reason + detailed cause + source-group + grid context present.
