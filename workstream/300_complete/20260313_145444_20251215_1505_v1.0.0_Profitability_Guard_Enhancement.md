# Gemini Coder - Profitability Guard Enhancement Plan (v1.0.0)

**Date**: 2025-12-15 15:05
**Version**: 1.0.0

## 1. Understanding of Requirements

The goal is to enhance the "Profitability Guard" in breakout_R_Rev.py to be more flexible and configurable via config.json. The current guard only checks if the cumulative P&L for a specific trade direction (net_acc > 0 or alt_acc > 0) is positive.

The new requirements introduce two additional, configurable criteria:

1.  **Sum of all today's trades for a selected group > Y**: Check if the sum of P&L for all trades (of a specific direction and type, e.g., 'net buy') closed *today* is greater than a configurable threshold `Y`.
2.  **Sum of most recent X trades for a selected group > Y**: Check if the sum of P&L for the most recent `X` trades (of a specific direction and type) is greater than a configurable threshold `Y`.

These options should apply to all four summary groups: `net_buy`, `net_sell`, `alt_buy`, and `alt_sell`. The configuration will be specific to each strategy name.

**Clarification**: The user has specified that all summary groups (`net_buy`, `net_sell`, `alt_buy`, `alt_sell`) for a given strategy will use the *same* set of rules (`mode`, `threshold_y`, `count_x`). The configuration will be unified per strategy, not per summary group.

## 2. Plan of Approach

The implementation will involve modifying breakout_R_Rev.py and establishing a new configuration structure in config.json.

1.  **Update `config.json`**: A new top-level `profitability_guard` section will be added. This section will contain nested configurations, where the first level key is the `script_name` (e.g., `breakout_R_Rev`). Directly under the `script_name` will be a single set of `mode`, `threshold_y`, and `count_x` parameters. These unified rules will apply to all four summary groups for that specific strategy.

    Example `config.json` structure:
    ```json
    "profitability_guard": {
      "breakout_R_Rev": {
        "mode": "all_time",  // "today", "recent_x"
        "threshold_y": 0,
        "count_x": 0         // Only relevant if mode is "recent_x"
      },
      "some_other_strategy": {
        "mode": "today",
        "threshold_y": 50
      }
    }
    ```
    For `breakout_R_Rev`, the default unified rule will be `"mode": "all_time", "threshold_y": 0`.

2.  **Create Helper Function (`_is_profitability_guard_passed`)**: A new private method, `_is_profitability_guard_passed(self, direction_key: str, mode_key: str) -> bool`, will be added to the `LiveTradingProcessor` class. This method will encapsulate the logic for evaluating the configured guard rules.

3.  **Implement Guard Logic**:
    *   The `_is_profitability_guard_passed` function will retrieve the *unified* profitability guard settings for the current `self.script_name` from `self.profitability_guard_config`.
    *   It will handle missing configuration by defaulting to the "all\_time" mode with a `threshold_y` of 0 (replicating the original behavior).
    *   Based on the `mode` specified in the unified configuration:
        *   **"today"**: It will filter `self.trade_log` for trades closed on the current day matching the `direction_key` (LONG/SHORT) and `mode_key` (net/alt), sum their P&L, and compare it against `threshold_y`.
        *   **"recent_x"**: It will filter `self.trade_log` for trades matching the `direction_key` and `mode_key`, take the `count_x` most recent ones, sum their P&L, and compare it against `threshold_y`.
        *   **"all_time"**: It will utilize the existing `self.cum_buy_net`, `self.cum_sell_net`, `self.cum_buy_alt`, `self.cum_sell_alt` accumulators and compare them against `threshold_y`.

4.  **Update `check_and_enter` Method**: The existing `if net_acc > 0:` and `if alt_acc > 0:` profitability guard checks in `check_and_enter` will be replaced with calls to the new `_is_profitability_guard_passed` helper function.
