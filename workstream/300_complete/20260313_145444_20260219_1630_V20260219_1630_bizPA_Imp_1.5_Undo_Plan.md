# Plan: Implementation 1.5 - Undo and Confirmation Logic [V20260219_1630]

**Project**: bizPA (Voice-First Small Trader Capture App)  
**Objective**: Implement a safety-first "Undo" mechanism using the audit log and refine the voice confirmation/execution state.  
**Version**: V20260219_1630

## Checklist
- [x] 1. Create Action Controller for Global Commands - 100%
- [x] 2. Implement Undo Logic (`POST /api/v1/action/undo`) - 100%
- [x] 3. Implement Reversion Logic (Insert -> Delete, Update -> Restore Old) - 100%
- [x] 4. Link "Undo" to Voice Intent (`undo_last_action`) - 100%
- [x] 5. Verify Undo across Create/Update/Archive actions - 100%

## Approach
1.  **Undo Mechanism**: The `/api/v1/action/undo` endpoint will identify the most recent `audit_events` record for a specific `device_id` within a 60-second window.
2.  **Reversion Logic**:
    - If the last action was `create`: Hard delete the entity.
    - If the last action was `update`: Restore the previous state from the `diff_log` (PostgreSQL `audit_events` stores the snapshot).
    - If the last action was `delete` (archived): Restore status to `confirmed` or `draft`.
3.  **Voice Integration**: Update the `voiceController` so that the `undo_last_action` intent internally triggers the undo logic and returns a TTS confirmation ("Action undone").
4.  **Audit Integrity**: Mark the audit record as "undone" to prevent double-undoing.

## Verification
- Create an item, call Undo, verify item is gone.
- Update an item, call Undo, verify fields reverted to original values.
- Delete an item, call Undo, verify status is restored.
- Verify voice command "Undo last action" results in successful reversion.
