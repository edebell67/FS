# Top X Workflow Engineering Plan

## Architecture Overview
The goal is to replicate the behavior of `profile_match_workflow` into a new, smaller, distinct continuous workflow called `top_x_multi_chart_workflow`. It extracts the absolute best `X` strategies directly from the backend Top 20 array based on specific Scalper/Rev-Scalper constraints, and forces them into the active Multi-Chart views.

## 1. Backend Modifications (`trade_viewer_api.py`)

- **Config Intialization**: Extend the `WORKFLOWS` global array to include a static definition for `top_x_multi_chart_workflow`:
  ```json
  {
      "id": "top_x_multi_chart_workflow",
      "name": "Top X Multi-Chart Loader",
      "description": "Extracts the Top X (by total net) strategies from the Top 20 list and forces them into the Multi-Chart view.",
      "enabled": false,
      "config": {
          "top_x": 6,
          "include_scalper": true,
          "include_rev_scalper": true
      },
      ...
  }
  ```
- **Execution Engine**: Add an `elif wf_id == "top_x_multi_chart_workflow":` dispatch path inside `_workflow_scheduler_loop()` and `run_workflow_now()`.
- **Implementation logic (`_run_top_x_multi_chart_workflow`)**:
  - Await/call `_get_top_20(mode, date_str)` to pull the already-computed best strategies of the day.
  - Pluck `top_x`, `include_scalper`, and `include_rev_scalper` from the user's saved config payload.
  - Implement a `is_scalper` and `is_rev_scalper` definition parser (already standard in the codebase: e.g. `< 10 TP, > 40 SL`).
  - Filter the Top 20 list depending on which booleans are checked. (If both are checked, include matches for *either* type).
  - Sort the filtered outputs exclusively by descending `Total Net`.
  - Slice (`[:top_x]`) to get exactly X elements (Top 1, Top 2, Top 5, etc).
  - Cast these items into the `{ strategy, product, group: "WF_TOPX_..." }` JSON format identical to the Profile Match payload.
  - Dump this to `multi_chart_payload.json` with `"source": "top_x_multi_chart_workflow"`.

## 2. Frontend Settings Engine (`workflow_automation.html`)

- In the massive string-template iterator (`render()`), add a conditional branch for our new `top_x_multi_chart_workflow` ID.
- Render exactly 3 core inputs:
  - `<input type="number" id="tx_wfid" value="{cfg.top_x}">` (Labeled "Top X Count", restricted mechanically to `max="20"`).
  - `<input type="checkbox" id="sc_wfid" checked="{cfg.include_scalper}">` (Labeled "Include Scalpers").
  - `<input type="checkbox" id="rev_wfid" checked="{cfg.include_rev_scalper}">` (Labeled "Include Rev Scalpers").
- In `saveWorkflow()`, explicitly harvest these three `document.getElementById` values.
- Formulate the JSON payload and POST it, ensuring the JSON format strictly honors numeric/boolean casting so Python parses it safely.

## 3. Frontend Multi-Chart Consumers (`multi_chart.js`, `multi_chart_v2.js`, `multi_chart_v3.js`)

- Hook into the `consumeSummaryImportPayload()` function.
- Add an `isTopXWorkflow = payload.source === 'top_x_multi_chart_workflow';` boolean flag next to the existing `isProfileWorkflow` flag.
- Implement the "Wipe" logic:
  - If `isTopXWorkflow` is true, immediately scrub the `activeOverlays` buffer clean of any legacy configurations prefixed with `WF_TOPX_`.
  - Scrub `localStorage.getItem('pnl_graph_views_multi')` of any old `WF_TOPX_...` presets.
- Let the pipeline naturally push the new payload items into `activeOverlays` and subsequently trigger the `updateCharts()` sequence.

### Validation Constraints
- By design, X can be any valid number but must logically cap at 20 because the underlying query mechanism (`_get_top_20`) fundamentally restricts the haystack to 20 elements.
