# Virtual Trade Close Pending Status - Prevent Unnecessary Churn

## Goal Description

Prevent virtual trades from being unnecessarily closed and immediately reopened when the leader changes temporarily. Instead of closing a trade immediately when it's no longer the leader, mark it as "close pending" and give it a chance to regain leadership. If the same product/direction becomes the leader again, revert to "open" status instead of creating a new trade.

**Problem:** Currently seeing rapid open/close cycles (198 virtual trades in one day) because trades are closed immediately when leadership changes, even if the same product/direction becomes leader again moments later.

**Solution:** Add a "close pending" intermediate status with smart reactivation logic.

## User Review Required

> [!IMPORTANT]
> This change introduces a new trade status: `CLOSE_PENDING`
> 
> **Status Flow:**
> - `OPEN` → `CLOSE_PENDING` (when leader changes)
> - `CLOSE_PENDING` → `OPEN` (if same product/direction regains leadership)
> - `CLOSE_PENDING` → `CLOSED` (if different product/direction becomes leader, or timeout expires)

> [!WARNING]
> Need to decide on timeout duration for `CLOSE_PENDING` status:
> - **Recommended:** 60-120 seconds (matches `vtrade_persistence_seconds`)
> - Too short: Won't prevent churn effectively
> - Too long: Delays actual leader changes

   ```python
   def _mark_v_trade_close_pending(trade_data: Dict, virtual_dir: Path) -> None:
       """Mark a virtual trade as close pending."""
       trade_data['status'] = 'CLOSE_PENDING'
       trade_data['close_pending_since'] = datetime.utcnow().isoformat()
       trade_data['close_pending_reason'] = 'AWAITING_NEW_LEADER'
       
       filepath = trade_data.get('filepath')
       if filepath:
           trade_data.pop('filepath', None)
           _safe_save_json(filepath, trade_data, indent=2)
   ```

2. **Add Reversion Logic (NEW - Critical!):**
   ```python
   def _revert_close_pending_to_open(trade_data: Dict, virtual_dir: Path) -> None:
       """Revert a close pending trade back to open when new candidate is rejected."""
       trade_data['status'] = 'OPEN'
       trade_data.pop('close_pending_since', None)
       trade_data.pop('close_pending_reason', None)
       trade_data['rejected_new_leader_count'] = trade_data.get('rejected_new_leader_count', 0) + 1
       trade_data['last_rejection_time'] = datetime.utcnow().isoformat()
       
       filepath = trade_data.get('filepath')
       if filepath:
           trade_data.pop('filepath', None)
           _safe_save_json(filepath, trade_data, indent=2)
   ```

3. **Add Same Product/Direction Check:**

   ```python
   def _is_same_product_direction(trade1: Dict, trade2: Dict) -> bool:
       """Check if two trades have the same product and direction."""
       return (trade1.get('product') == trade2.get('product') and
               trade1.get('direction') == trade2.get('direction'))
   ```

4. **Update Main Loop Logic:**

   ```python
   # In _manage_virtual_trades for single leader mode:
   
   # Get current state

   # Determine new leader candidate
   ```

**Scenario 1: Quick Leader Flip-Flop**
```
1. GBPEUR_C BUY is leader → Status: OPEN
2. CAD SELL becomes leader → GBPEUR_C BUY Status: CLOSE_PENDING
3. GBPEUR_C BUY becomes leader again (within 120s) → Status: OPEN (reactivated!)
```

**Scenario 2: Sustained Leader Change**
```
1. GBPEUR_C BUY is leader → Status: OPEN
2. CAD SELL becomes leader → GBPEUR_C BUY Status: CLOSE_PENDING
3. CAD SELL remains leader for >120s → GBPEUR_C BUY Status: CLOSED
```

**Scenario 3: Different Product/Direction**
```
1. GBPEUR_C BUY is leader → Status: OPEN
2. GBPAUD_C SELL becomes leader → GBPEUR_C BUY Status: CLOSE_PENDING
3. GBP SELL becomes leader → GBPEUR_C BUY Status: CLOSED (different product)
                            → GBPAUD_C SELL Status: CLOSED (different product)
```
