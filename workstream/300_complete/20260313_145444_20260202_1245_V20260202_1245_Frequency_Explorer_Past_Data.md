# Plan: Frequency Explorer - Disable Past Data Refresh

## 1. Understanding of Requirements
The user wants to prevent the refreshing of "Snapshot Timeline" data in `/frequency_explorer.html` if the data belongs to the past (i.e., a historical date is selected). "Old data must be kept as is".

Currently, `fetchData()` fetches the JSON file. If a historical date is selected via `selectedDate`, it constructs a path like `json/live/YYYY-MM-DD/_frequency.json`. If auto-refresh is on, it keeps re-fetching this file.

The user's request implies that historical files (files for past dates) are static and shouldn't change, so we shouldn't waste resources or cause UI re-renders by auto-refreshing them. Or perhaps the "auto-refresh" logic itself should be disabled when viewing a past date.

## 2. Plan of Approach
1.  **Modify `fetchData` in `frequency_explorer.html`**:
    *   No changes strictly needed here to *prevent* the fetch if we control the trigger, but I should ensure that if I am viewing a historical date, I might not need to fetch repeatedly.
    *   However, the user says "ensure that the data in a Snapshot Timeline that is in the past are not refreshed". This could mean:
        *   a) Disable auto-refresh when a past date is selected.
        *   b) If `selectedDate` < Today, simply don't fetch again if we already have data?
        *   c) The user might be referring to the *content* of the snapshot timeline. "Snapshot Timeline" is a specific section.

    Let's assume the most robust interpretation: **Disable Auto-Refresh when viewing a historical date.**
    If `selectedDate` is set and represents a past date (not today), the data is static (market closed or day over). Repeated fetching is pointless.

2.  **Implementation Details**:
    *   In `applyDateFilter()`, check if the selected date is in the past. If so, stop the auto-refresh timer and unchecked the box (or just pause the timer internally).
    *   In `setupAutoRefresh()`: Add a check inside the interval. If `selectedDate` is historical, do not fetch.
    *   But wait, `selectedDate` is only set when filtering. If `selectedDate` is null (Latest), we assume today/live.
    *   Logic:
        *   Get current date (YYYY-MM-DD).
        *   If `selectedDate` is defined and `selectedDate` != current date:
            *   It's a past/historical view.
            *   Disable auto-refresh logic.

3.  **Refining the Requirement "Snapshot Timeline that is in the past"**:
    *   The user might be talking about specific *snapshots* within the timeline? "data in a Snapshot Timeline that is in the past".
    *   The timeline displays *snapshots*. Snapshots are historical by definition (they are captured points in time).
    *   "old data must be kept as is" -> This sounds like "don't update the UI for historical snapshots even if the file changes" OR "don't re-fetch the file if it's a past date".
    *   Given the file structure `_frequency.json` grows by appending snapshots, fetching it again just adds new snapshots. It doesn't usually change old ones.
    *   I will stick to the "Disable Auto-Refresh for historical dates" interpretation as it's the safest way to "keep old data as is" (by not reloading the file).

4.  **Action Plan**:
    *   Update `setupAutoRefresh` to check if we are in "Historical Mode".
    *   Helper function `isHistoricalView()`: returns true if `selectedDate` is set and not today.
    *   Update `initExplorer` or `applyDateFilter` to toggle the Auto Refresh UI based on this state.
