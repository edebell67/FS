# Fix Live Trade Counting Logic

## 1. Issue
The user noticed `max_live_trades` (set to 1) was exceeded (4 concurrent trades).
Investigation revealed a logic flaw in `common.py`:
- `max_live_trades` is enforced by `_create_l_trade_order`, which calls `_get_total_open_live_trades`.
- `_get_total_open_live_trades` counts trades where `"is_live_trade": true`.
- However, when a trade sends an order (`order_sent_net: true`), the `is_live_trade` flag is NOT immediately set to `true` in memory or disk in all paths (it relies on finalization or specific activation triggers).
- As a result, newly opened live trades are ignored by the counter until they are "finalized" or explicitly promoted, allowing subsequent trades to bypass the limit.

## 2. Requirement
Ensure `max_live_trades` is strictly enforced by correctly counting ALL active live trades, including those that have sent orders but might not have the `is_live_trade` flag set yet.

## 3. Plan of Approach
1.  **Modify `common.py`**:
    *   Update `_get_total_open_live_trades`.
    *   Expand the counting condition to include trades where `order_sent_net` or `order_sent_alt` is True, in addition to `is_live_trade`.
2.  **Verify**:
    *   The logic is based on code analysis. Verification will be by ensuring the code reflects the new inclusive condition.
3.  **Update Version**:
    *   Update `constants.py` to `V20260203_1100`.
