# Plan to Fix PnL Initialization Logic

## 1. Issue Description
The user pointed out that `threshold_y` is configurable (currently 0, but could be negative).
However, `top_buy_pnl` and `top_sell_pnl` are currently initialized to `-1`.
This prevents the system from identifying the "top" strategy if the best available strategy has a PnL e.g. `-50` and the threshold is `-100`. The loop condition `stats['pnl'] > top_buy_pnl` (i.e. `-50 > -1`) would be false, so it wouldn't update, effectively ignoring valid strategies in loss-tolerant configurations.

## 2. Plan of Approach
I will initialize `top_buy_pnl` and `top_sell_pnl` to negative infinity (`-float('inf')`) instead of `-1`.
This ensures that the *first* strategy encountered (or any subsequent better one) will always be captured as the initial "top" candidate, regardless of how negative its PnL is. The actual filtering against `guard_threshold_y` happens *later* in the reconciliation block, which is the correct place for it.
