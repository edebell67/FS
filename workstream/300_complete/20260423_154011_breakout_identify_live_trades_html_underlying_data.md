## Findings
`fs/live_trades.html` uses two direct data sources:

1. `GET /api/trade_file`
   - Called as:
     - `/api/trade_file?mode=<run_mode>&date=<date>&filename=_live_trades.json&product_type=<product_type>`
   - Used by:
     - `loadLiveTrades()`
   - Purpose:
     - Loads the aggregate L-trade artifact for the selected `mode/date/product_type`.

2. `GET /api/config`
   - Used by:
     - `loadConfig()`
   - Purpose:
     - Supplies UI/runtime settings such as:
       - `run_mode`
       - `product_type` / `product_types`
       - `max_live_trades`
       - `daily_target`

## Underlying Data Lineage
Primary page data:
- `TradeApps/breakout/fs/json/<mode>/<product_type>/<date>/_live_trades.json`

How `_live_trades.json` is built:
- Generator:
  - `TradeApps/breakout/fs/extract_live_trades.py`
- Input files scanned in the day folder:
  - `*_op.json` => treated as `OPEN`
  - `*_cld.json` => treated as `CLOSED`
- Only records with:
  - `is_live_trade == true`
  - are included.
- Output payload shape:
  - `generated_at`
  - `trade_count`
  - `trades: [...]`

Fields surfaced into the page come from those underlying trade files, including:
- `trade_id`
- `product`
- `script_name` / `app_name`
- `source_strategy` / `strategy`
- `direction`
- `entry_time`
- `exit_time`
- `entry_price`
- `exit_price`
- `net_return`
- `status`
- `source_group` / `source_screen`
- `is_live_trade`

Aggregation behavior:
- `extract_live_trades.py` uses a canonical key per source file stem and script name.
- If both open and closed versions exist for the same canonical trade, the closed record can replace the open one in the aggregate view.

## API-Side Behavior
`/api/trade_file` is generic, but for this page it does one important extra step before reading `_live_trades.json`:
- calls `_refresh_open_trade_files(run_mode, date, product, product_type)`

That refresh path:
- loads configured `trade_products`
- fetches latest quotes
- calls `common._update_open_trade_json_prices(...)`
- rewrites persisted OPEN trade JSON files with refreshed:
  - `current_price`
  - open PnL / derived values

Important consequence:
- `live_trades.html` does not compute live PnL from market data itself.
- It displays whatever is already persisted into the underlying trade JSON files and then aggregated into `_live_trades.json`.

## Operational Source
The aggregate file is maintained by the running app loop:
- `TradeApps/breakout/fs/verify_algo_execution_and_restart_02.py`
- includes:
  - `extract_live_trades.py`

So the page’s main dataset is not generated on-demand in the browser. It is a persisted artifact kept current by the background script loop.

## End-to-End Mapping
- Page controls:
  - `live_trades.html`
- Config/UI defaults:
  - `/api/config`
  - `TradeApps/breakout/fs/config.json`
- Trade table rows:
  - `/api/trade_file?...filename=_live_trades.json...`
  - `json/<mode>/<product_type>/<date>/_live_trades.json`
- `_live_trades.json` source inputs:
  - day-folder `*_op.json`
  - day-folder `*_cld.json`
- OPEN trade value refresh before read:
  - `trade_viewer_api.py` `_refresh_open_trade_files`
  - `common.py` `_update_open_trade_json_prices`
