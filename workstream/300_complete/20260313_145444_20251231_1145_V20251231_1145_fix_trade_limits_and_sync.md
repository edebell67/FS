# Plan: Fix Max Open Trades Guard and Global State Sync

This document outlines the plan to fix the `max_live_trades` guard logic and ensure the global active trades counter stays in sync.

## 1. Understanding of Requirements

*   **Guard Failure**: The `order_sent_net` and `order_sent_alt` flags are being set to `True` even if the `max_live_trades` guard blocks the order creation. This populates JSON files with incorrect "live" status.
*   **State Desync**: The `active_trades.json` file is bloated with incorrect counts (e.g., 50+) because it wasn't cleared after unauthorized trades were converted back, and there are likely over-increments during script restarts.
*   **Race Conditions**: Multiple processes (scripts) checking the same JSON file simultaneously might bypass the limit.

## 2. Plan of Approach

1.  **Modify `common.py`**:
    *   Add `ACTIVE_TRADES_LOCK_FILE` and implement a simple locking mechanism for `_increment` and `_decrement` to prevent race conditions.
    *   Update `_handle_live_orders`: Only set `order_sent` flags if `_create_tradeable_json` returns a successful path.
    *   Capture "Max live trades reached" as a `trade_block_reason`.
    *   Optimize `_load_persisted_state` to ensure it doesn't double-count on restart (idempotent increment).
2.  **Create `sync_active_trades.py`**:
    *   A utility to scan the `orders` directory (from `config.json`) and rebuild `active_trades.json` based on actual open order files found.
3.  **Run Sync**: Execute the sync script to fix the current broken state.
4.  **Update Version**: Bump to `V20251231_1145`.
