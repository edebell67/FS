# Rule #6 (gen on neg alt_net close) SQL-backed plan

## Background
- Current implementation keeps an in-memory deque of closed-trade summaries (`_record_closed_trade_summary`).
- `_should_generate_alt_net_json_on_open` only scans that deque for `alt < 0` entries to decide whether the next open is a 'C' trade.
- We want to replace that memory dependency with a query against dbo.tbl_105_trade_lifecycle_snapshots/its view so all panels share the same truth and we can recover state after restarts.

## Requirements from user
1. Use the dbo.tbl_105_trade_lifecycle_snapshots table (or exposed view) instead of Python deques.
2. Filter rows where event_type = 'close' and the session_id/screen_session_name matches the current UI session (embedded in the instance name).
3. Restrict to rows whose timestamp falls on the current date.
4. Apply TOP (lookback) where lookback equals the existing s09_lookback_var (validated via _get_csv_rule_lookback).
5. Return true if any of those rows has alt_pnl < 0.
6. Example SQL provided by user is:
   SELECT CASE WHEN EXISTS (
       SELECT 1
       FROM (
           SELECT TOP (2)
               alt_pnl
           FROM tradedb.dbo.tbl_105_trade_lifecycle_snapshots
           WHERE event_type = 'close'
             AND session_id LIKE '%8d0b341b%'
           ORDER BY timestamp DESC
       ) AS x
       WHERE x.alt_pnl < 0
   ) THEN 1 ELSE 0 END AS result;
   We must adapt this to use the actual session identifier (no hard-coded pattern) and TOP (lookback).

## Proposed implementation steps
1. Session sourcing
   - TradeManager already has self.session_guid (UUID per screen instance).
   - Expose a helper to map this to the session filter string. If the DB stores the full session string, use equality; if it stores prefixes, preserve the ability to do LIKE '%short%' (confirm via schema inspection).

2. DB query helper
   - Create a method (e.g., _has_negative_alt_close_recently_via_db(product, trade_type, lookback) -> bool).
   - Build a SQL command matching the user’s template, parameterizing: session ID, event type close, TOP (lookback), and timestamp >= today 00:00.
   - Execute it via an existing DB access path: if we have a shell-accessible view command, call it through subprocess or sqlcmd. Handle network errors gracefully and log them.

3. Hook into rule #6
   - Update _should_generate_alt_net_json_on_open to call the new helper when csv_gen_rule_var == 'gen on neg alt_net close'.
   - Keep _get_csv_rule_lookback for range validation, but skip the deque scan entirely for this rule.
   - Retain the current fallback for other rules.

4. Fallback / resilience
   - If the SQL query fails, log the error and default to False (no 'C' trigger) to avoid spurious trades.
   - Optionally keep the deque logic as a fallback if the DB is unavailable (document whichever choice we make).

5. Testing / verification
   - Add a test or script harness that mocks the DB helper to return True/False and ensures _should_generate_alt_net_json_on_open honors it.
   - Manual verification: run the SQL query in the shell with a known session ID and lookback to confirm expected results.

## Open questions
- Exact column name (session_id vs screen_session_name) and whether it already stores the GUID or a composite string.
- Whether timestamps in the table are UTC or local; we need to match “today” accordingly.
- Preferred DB access method from the app (direct pyodbc/SQLAlchemy vs shell sqlcmd).
