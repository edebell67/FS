# Epic Decomposition

When given an epic document, decompose it into individual task files that can be executed autonomously by AI agents. Each task must contain enough context and ordering metadata for a model to decide whether it is ready to start.

Preserve source text fidelity when copying epic metadata:
- Keep UTF-8 punctuation and symbols from the source epic title and body.
- Do not replace em dashes, smart quotes, or accented characters with `?`, `�`, or ASCII fallbacks unless the source file already contains corruption.

## Input

An epic markdown file containing:
- Objective or goal
- System architecture, optional
- Workstreams labeled like `WORKSTREAM A - DATA LAYER`
- Tasks within each workstream, such as `TASK A1 Define publishable signal schema`

## Output

Individual task files in the workstream's designated agent folder:

```text
workstream/100_backlog/{agent}/
|- {timestamp}_{epic}_workstream{X}_{task_name}.md
|- {timestamp}_{epic}_workstream{X}_{task_name}.md
'- ...
```

Example for `Autonomous Trading Signal Platform`:

```text
workstream/100_backlog/gemini/
|- 20260309_120000_autonomous_trading_signal_platform_workstreamA_define_publishable_signal_schema.md
|- 20260309_120001_autonomous_trading_signal_platform_workstreamA_define_publishable_trade_result_schema.md
'- ...
```

## File Naming Convention

```text
{yyyymmdd}_{hhmmss}_{epic_snake_case}_workstream{Letter}_{task_name_snake_case}.md
```

Rules:
- Include the epic name in snake_case after the timestamp.
- Use a consistent timestamp base for all tasks in one decomposition batch.
- Increment seconds to preserve sort order within a workstream.
- Use snake_case for all file-name components.
- Keep task names concise but descriptive.
- Ensure the epic name matches the `Epic:` metadata inside the task file.

## Task File Template

Each generated task file must follow this structure:

```markdown
# TASK {ID}: {Task Title}

**Workstream:** {Letter} - {WORKSTREAM NAME}
**Epic:** {Epic Title}
**Epic Sequence:** {sequence_number}
**Depends On:** {sequence numbers or `none`}
**Blocks:** {sequence numbers or `none`}
**Readiness:** {ready | blocked}
**Status:** [ ] Not Started

---

## Purpose

{Why this task exists - extracted from epic or inferred}

## Input

{What this task requires as input - data, dependencies, prior tasks}

## Output

{What this task produces - files, services, artifacts}

## Fields / Components (if applicable)

- {field_1}
- {field_2}

## Action (if applicable)

{Specific action to perform}

## Verification

- [ ] {Verification criteria from epic}
- [ ] {Additional verification if needed}

---

## Notes

_Add implementation notes here_
```

Ordering field rules:
- `Epic Sequence` identifies the dependency layer within the epic, such as `1.1`, `1.2`, or `2.1`.
- `Depends On` lists predecessor epic sequence numbers when known; otherwise set `none`.
- `Blocks` lists downstream tasks only when the epic explicitly identifies them; otherwise set `none`.
- `Readiness` is `ready` only when no predecessor work is required at task creation time.
- If dependency edges are ambiguous, tell the assignee to use `$task-execution-ordering` before implementation starts.
- Foundation tasks that establish schemas, contracts, shared models, migrations, or canonical enums must explicitly declare what they unblock.
- If sibling tasks name a foundational task as input or dependency, the foundational task must not omit its downstream `Blocks` metadata unless the decomposition engine truly cannot infer the affected tasks.

## Decomposition Process

### Step 1: Parse Epic Structure

Identify all workstreams by looking for patterns:
- `# WORKSTREAM {LETTER} - {NAME}`
- `## WORKSTREAM {LETTER} - {NAME}`
- Numbered sections with clear boundaries

### Step 2: Extract Tasks

For each workstream, identify tasks by patterns:
- `## TASK {ID} {Title}`
- `### TASK {ID} {Title}`
- Numbered task lists

Extract from each task:

| Field | Source Pattern |
|-------|----------------|
| Purpose | Purpose section or inferred from context |
| Input | Input section |
| Output | Output section |
| Fields | Fields section, usually a bulleted list |
| Action | Action section |
| Verification | Verification section |
| Epic Sequence | Dependency layer inferred from epic ordering |
| Depends On | Explicit prerequisites or inferred predecessor tasks |
| Blocks | Explicit downstream tasks, if stated |
| Readiness | `ready` if no predecessors, otherwise `blocked` |
| Status | Default to `[ ] Not Started` |

Task-quality requirements during extraction:
- Add a `Dependencies` section only when it improves execution clarity, but never omit dependency metadata in the header.
- For foundational tasks, explicitly capture downstream consumers in `Blocks` when they are inferable from sibling tasks.
- Make `Output` concrete. Prefer deliverable-shaped artifacts such as schema files, enum definitions, service contracts, field maps, migration specs, API endpoints, UI screens, or test suites over vague phrases like "logic" or "definitions".
- Keep verification bullets implementation-facing and completion-oriented. Avoid generic restatements of purpose.
- Deduplicate verification bullets after merging requirements from the epic and any auxiliary skills.
- When a UI skill expands verification requirements, keep one canonical bullet per requirement. Do not repeat startup-script, localhost URL, smoke-test, or screenshot requirements with alternate wording.

### Step 3: Generate Task Files

For each extracted task:
1. Generate the filename using the naming convention.
2. Populate the template with extracted data and ordering metadata.
3. Write the file to the appropriate agent folder based on workstream assignment.

Assign ordering metadata during decomposition:
1. Build a dependency graph for tasks inside the same epic using explicit dependencies first.
2. Infer missing edges from phase order, required outputs, schema prerequisites, migration-before-consumer rules, and validation gates.
3. Assign `Epic Sequence` by dependency layer.
   - Parallel foundation tasks: `1.1`, `1.2`, `1.3`
   - Tasks waiting on layer 1: `2.1`, `2.2`
   - Continue incrementing major layers as dependencies deepen
4. Set `Readiness: ready` only for tasks with `Depends On: none`.
5. Set `Readiness: blocked` for tasks that have unresolved predecessors at creation time.
6. When uncertainty remains, keep the more conservative dependency set and add a note telling the assignee to verify ordering with `$task-execution-ordering`.

Before finalizing each generated task file:
1. Check that `Input`, `Output`, `Dependencies`, `Depends On`, `Blocks`, and `Verification` are mutually consistent.
2. If another task depends on this task's outputs, ensure this task's `Blocks` metadata names those downstream layers or tasks where inferable.
3. Rewrite vague outputs into explicit artifacts or artifact families.
4. Remove duplicate verification bullets and collapse overlapping requirements into one stronger bullet.
5. Confirm copied epic metadata still matches the original source text byte-for-byte for visible punctuation and title wording.

### Step 4: Organize by Agent

Distribute workstreams to agent folders based on assignment:

```text
workstream/100_backlog/
|- gemini/      # Assigned workstreams
|- claude/      # Assigned workstreams
|- codex/       # Assigned workstreams
'- general/     # Unassigned or cross-cutting tasks
```

## Integration with Agent Controller

Generated task files are automatically picked up by `run_agent.ps1`:

1. The agent polls `100_backlog/{agent}`.
2. It reads the task file, extracts `Epic:`, ordering metadata, and `Epic Output Folder` metadata.
3. It writes code and deliverable artifacts to the `Epic Output Folder` path.
4. It executes the task with full context.
5. It moves the task file to `300_complete` or `400_failed`.

Important:
- Task files remain in the `workstream` folders.
- Code, tests, and deliverable artifacts must be written to the `Epic Output Folder` specified in the task metadata.

## Epic Augmentation

When an epic has been decomposed and its solution is partially or fully implemented, additional tasks may be needed to fill gaps or add new functionality. Epic augmentation generates new tasks that build ON TOP OF the existing solution.

### When to Use Augmentation

Use augmentation when:
- Original decomposition is missing key elements (no UI, no tests, no docs, no infrastructure)
- New features need to be added to an existing solution
- Coverage gaps are identified during review
- Solution needs expansion based on user feedback

### Augmentation vs New Decomposition

| Scenario | Action |
|----------|--------|
| Epic has no solution folder | New decomposition |
| Epic has solution folder + missing components | Augmentation |
| Adding new feature to existing epic | Augmentation |
| Complete rework of epic scope | New decomposition |

### Augmentation Types

| Type | Description | Output Path |
|------|-------------|-------------|
| UI | Create frontend for existing backend API | `ep_{slug}/solution/frontend/` |
| Tests | Add test suites for existing code | `ep_{slug}/solution/{backend,frontend}/tests/` |
| Docs | Create README, API docs, usage guides | `ep_{slug}/` |
| Infra | Add setup scripts, Docker, CI/CD | `ep_{slug}/` |
| Feature | New functionality building on existing code | Varies by feature scope |

### Solution Analysis

Before generating augmentation tasks, the system analyzes the existing solution folder to understand:

1. **Structure Detection**

   - Identify data models consumed

# UI Delivery Viewability

When delivering a task that involves creating, updating, or launching a user interface (UI), web page, screen, dashboard, or frontend component, you MUST ensure the user can easily view and interact with the delivery locally without having to manually figure out how to start the required services.

This skill mandates the creation of helper scripts and the validation of local environments to guarantee seamless UI testing.

## Core Requirements

1. **Provide a Starter Script**:
   Whenever a UI is the primary deliverable, you must create or update a simple executable script (e.g., a `.bat`, `.ps1`, or `.sh` file) in the project root or relevant directory that starts all necessary backend and frontend services. 
   - Example: `start_dashboard.bat` or `run_ui.bat`.
   - The script must clearly output the local URL (e.g., `http://localhost:8000`) where the UI can be accessed.

2. **Validate Environment Dependencies**:
   Before finalizing the delivery or asking the user to verify the UI, check for common environment issues that cause immediate crashes:
   - Ensure requisite packages (like `uvicorn`, `python-multipart`, `fastapi`, etc.) are either documented in a `requirements.txt` / `package.json` or explicitly installed.
   - Run a quick smoke test of the startup sequence yourself to ensure it doesn't immediately crash due to missing dependencies.

3. **Handle Blocking Errors Gracefully**:
   If the UI depends on a database connection (like PostgreSQL) or an external API that might not be available during initial local testing, ensure the application startup is resilient. 
   - E.g., catch missing database connection errors at startup and allow the UI to serve generic or cached data, rather than crashing the entire web server.

4. **Document the Access Method in Checklist**:
   In your task execution evidence, explicitly state the command or script the user should click/run, and provide the exact `localhost` URL they should open in their browser to verify the task.

5. **Capture Screenshot Evidence**:
   After successfully launching the UI, you MUST capture a screenshot as proof of delivery:
   - Use an automated screenshot tool (e.g., Playwright, Puppeteer, or system screenshot utility)
   - Save screenshot to the epic's `verification/` folder with naming: `{task_id}_screenshot.png`
   - If multiple screens/states exist, capture each: `{task_id}_screenshot_01.png`, `{task_id}_screenshot_02.png`, etc.
   - Screenshot must show the UI in a working state (not error pages)

6. **Include Access Script in Evidence**:
   The starter script itself is part of the deliverable evidence:
   - Copy or reference the access script path in the verification output
   - Include the full script content in the task result markdown
   - Format example in result:
     ```
     ## Access Script
     Path: C:\path\to\start_ui.bat

     ```bat
     @echo off
     echo Starting UI...
     python -m uvicorn main:app --port 8000
     ```

     Run this script and open http://localhost:8000
     ```
   - If the UI requires specific environment setup (env vars, config files), document those in the evidence

## Screenshot Automation

Use one of these methods to capture screenshots:

### Option 1: Playwright (Recommended)
```python
# screenshot_capture.py
from playwright.sync_api import sync_playwright

def capture_screenshot(url, output_path):
    with sync_playwright() as p:
        browser = p.chromium.launch()
        page = browser.new_page(viewport={'width': 1920, 'height': 1080})
        page.goto(url)
        page.wait_for_load_state('networkidle')
        page.screenshot(path=output_path, full_page=True)
        browser.close()

capture_screenshot('http://localhost:8000', 'verification/task_screenshot.png')
```

### Option 2: PowerShell (Windows)
```powershell
# capture_screenshot.ps1
Add-Type -AssemblyName System.Windows.Forms
Start-Process "http://localhost:8000"
Start-Sleep 5
[System.Windows.Forms.Screen]::PrimaryScreen | ForEach-Object {
    $bitmap = New-Object System.Drawing.Bitmap($_.Bounds.Width, $_.Bounds.Height)
    $graphics = [System.Drawing.Graphics]::FromImage($bitmap)
    $graphics.CopyFromScreen($_.Bounds.Location, [System.Drawing.Point]::Empty, $_.Bounds.Size)
    $bitmap.Save("verification\task_screenshot.png")
}
```

### Option 3: Browser DevTools Protocol

## Why this is important
UI tasks require visual verification. If the underlying server crashes due to missing dependencies (like a missing `python-multipart` library) or lacking startup scripts, the user wastes time debugging the environment rather than reviewing your design. Always make UI deliverables instantly testable.

Screenshot evidence provides:
- Proof the UI was successfully rendered
- Visual record for future reference
- Baseline for regression testing
- Documentation for stakeholders who cannot run locally
