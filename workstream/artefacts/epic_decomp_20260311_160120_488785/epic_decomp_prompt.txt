# Epic Decomposition

When given an epic document, you MUST decompose it into individual task files that can be executed autonomously by AI agents. Each task becomes a standalone markdown file with all context needed for independent execution.

## Input

An epic markdown file containing:
- Objective / Goal
- System architecture (optional)
- Workstreams (labeled sections, e.g., "WORKSTREAM A — DATA LAYER")
- Tasks within each workstream (e.g., "TASK A1 Define publishable signal schema")

## Output

Individual task files in the workstream's designated agent folder:
```
workstream/100_todo/{agent}/
├── {timestamp}_{epic}_workstream{X}_{task_name}.md
├── {timestamp}_{epic}_workstream{X}_{task_name}.md
└── ...
```

Example for "Autonomous Trading Signal Platform" epic:
```
workstream/100_todo/gemini/
├── 20260309_120000_autonomous_trading_signal_platform_workstreamA_define_publishable_signal_schema.md
├── 20260309_120001_autonomous_trading_signal_platform_workstreamA_define_publishable_trade_result_schema.md
└── ...
```

## File Naming Convention

```
{yyyymmdd}_{hhmmss}_{epic_snake_case}_workstream{Letter}_{task_name_snake_case}.md
```

**Examples:**
- `20260309_120000_autonomous_trading_signal_platform_workstreamA_define_publishable_signal_schema.md`
- `20260309_120010_autonomous_trading_signal_platform_workstreamB_create_sync_configuration.md`
- `20260309_120041_autonomous_trading_signal_platform_workstreamE_implement_x_connector.md`

**Rules:**
- Include epic name in snake_case after timestamp (enables filtering/grouping by epic)
- Use consistent timestamp base for all tasks in a decomposition batch
- Increment seconds to maintain sort order within workstream
- Use snake_case for all name components (lowercase, underscores)
- Keep task names concise but descriptive
- Epic name should match the `Epic:` metadata field inside the task file

## Task File Template

Each generated task file MUST follow this structure:

```markdown

**Workstream:** {Letter} — {WORKSTREAM NAME}
**Epic:** {Epic Title}

---

## Purpose

{Why this task exists - extracted from epic or inferred}

## Input

{What this task requires as input - data, dependencies, prior tasks}

## Output

{What this task produces - files, services, artifacts}

## Fields / Components (if applicable)

- {field_1}
- {field_2}
- ...

## Action (if applicable)

{Specific action to perform}

## Verification

- [ ] {Verification criteria from epic}
- [ ] {Additional verification if needed}

---

## Notes

_Add implementation notes here_
```

## Decomposition Process

### Step 1: Parse Epic Structure

Identify all workstreams by looking for patterns:
- `# WORKSTREAM {LETTER} — {NAME}`
- `## WORKSTREAM {LETTER} — {NAME}`
- Numbered sections with clear boundaries

### Step 2: Extract Tasks

For each workstream, identify tasks by patterns:
- `## TASK {ID} {Title}`
- `### TASK {ID} {Title}`
- Numbered task lists

Extract from each task:
| Field | Source Pattern |
|-------|----------------|
| Purpose | "Purpose" section or inferred from context |
| Input | "Input" section |
| Output | "Output" section |
| Fields | "Fields" section (bulleted list) |
| Action | "Action" section |
| Verification | "Verification" section |
| Status | Default to `[ ] Not Started` |

### Step 3: Generate Task Files

For each extracted task:
1. Generate filename using naming convention
2. Populate template with extracted data
3. Write to appropriate agent folder based on workstream assignment

### Step 4: Organize by Agent

Distribute workstreams to agent folders based on assignment:

```
workstream/100_todo/
├── gemini/      # Assigned workstreams
├── claude/      # Assigned workstreams
├── codex/       # Assigned workstreams
└── general/     # Unassigned tasks
```

# UI Delivery Viewability

When delivering a task that involves creating, updating, or launching a user interface (UI), web page, screen, dashboard, or frontend component, you MUST ensure the user can easily view and interact with the delivery locally without having to manually figure out how to start the required services.

This skill mandates the creation of helper scripts and the validation of local environments to guarantee seamless UI testing.

## Core Requirements

1. **Provide a Starter Script**:
   Whenever a UI is the primary deliverable, you must create or update a simple executable script (e.g., a `.bat`, `.ps1`, or `.sh` file) in the project root or relevant directory that starts all necessary backend and frontend services. 
   - Example: `start_dashboard.bat` or `run_ui.bat`.
   - The script must clearly output the local URL (e.g., `http://localhost:8000`) where the UI can be accessed.

2. **Validate Environment Dependencies**:
   Before finalizing the delivery or asking the user to verify the UI, check for common environment issues that cause immediate crashes:
   - Ensure requisite packages (like `uvicorn`, `python-multipart`, `fastapi`, etc.) are either documented in a `requirements.txt` / `package.json` or explicitly installed.
   - Run a quick smoke test of the startup sequence yourself to ensure it doesn't immediately crash due to missing dependencies.

3. **Handle Blocking Errors Gracefully**:
   If the UI depends on a database connection (like PostgreSQL) or an external API that might not be available during initial local testing, ensure the application startup is resilient. 
   - E.g., catch missing database connection errors at startup and allow the UI to serve generic or cached data, rather than crashing the entire web server.

4. **Document the Access Method in Checklist**:
   In your task execution evidence, explicitly state the command or script the user should click/run, and provide the exact `localhost` URL they should open in their browser to verify the task.

5. **Capture Screenshot Evidence**:
   After successfully launching the UI, you MUST capture a screenshot as proof of delivery:
   - Use an automated screenshot tool (e.g., Playwright, Puppeteer, or system screenshot utility)
   - Save screenshot to the epic's `verification/` folder with naming: `{task_id}_screenshot.png`
   - If multiple screens/states exist, capture each: `{task_id}_screenshot_01.png`, `{task_id}_screenshot_02.png`, etc.
   - Screenshot must show the UI in a working state (not error pages)

6. **Include Access Script in Evidence**:
   The starter script itself is part of the deliverable evidence:
   - Copy or reference the access script path in the verification output
   - Include the full script content in the task result markdown
   - Format example in result:
     ```
     ## Access Script
     Path: C:\path\to\start_ui.bat

     ```bat
     @echo off
     echo Starting UI...
     python -m uvicorn main:app --port 8000
     ```

     Run this script and open http://localhost:8000
     ```
   - If the UI requires specific environment setup (env vars, config files), document those in the evidence

## Screenshot Automation

Use one of these methods to capture screenshots:

### Option 1: Playwright (Recommended)
```python
# screenshot_capture.py
from playwright.sync_api import sync_playwright

def capture_screenshot(url, output_path):
    with sync_playwright() as p:
        browser = p.chromium.launch()
        page = browser.new_page(viewport={'width': 1920, 'height': 1080})
        page.goto(url)
        page.wait_for_load_state('networkidle')
        page.screenshot(path=output_path, full_page=True)
        browser.close()

capture_screenshot('http://localhost:8000', 'verification/task_screenshot.png')
```

### Option 2: PowerShell (Windows)
```powershell
# capture_screenshot.ps1
Add-Type -AssemblyName System.Windows.Forms
Start-Process "http://localhost:8000"
Start-Sleep 5
[System.Windows.Forms.Screen]::PrimaryScreen | ForEach-Object {
    $bitmap = New-Object System.Drawing.Bitmap($_.Bounds.Width, $_.Bounds.Height)
    $graphics = [System.Drawing.Graphics]::FromImage($bitmap)
    $graphics.CopyFromScreen($_.Bounds.Location, [System.Drawing.Point]::Empty, $_.Bounds.Size)
    $bitmap.Save("verification\task_screenshot.png")
}
```

### Option 3: Browser DevTools Protocol
```bash
# Using Chrome headless
chrome --headless --screenshot="verification/task_screenshot.png" --window-size=1920,1080 http://localhost:8000
```

# 1. Product Summary

bizPA is a voice-first business capture and tax-readiness platform for sole traders and micro businesses.

Its primary purpose is not bookkeeping and not direct tax filing.

Its core promise is:

# 9. Functional Area 1: Capture & Input

## 9.1 Objective

Allow users to capture business activity continuously with minimum friction, especially through voice.

## 9.2 Supported Inputs

- voice entry
- manual form entry
- attachment/image upload
- notes
- invoice creation
- receipt/expense creation
- quote creation
- payment capture
- booking capture
- non-financial reminders or notes

## 9.3 Capture Philosophy

Everything starts from capture.

The system should interpret the input, structure it, and place it correctly without forcing the user to think in folders first.

## 9.4 Voice-First Direction

Voice must be a core path, not a novelty.

Examples:

- “Invoice John £500 plus VAT for guttering.”
- “Fuel receipt £60 including VAT.”
- “Quote Sarah £1,200 for bathroom repair.”
- “Payment received £600 from John.”
- “Note: call supplier tomorrow.”

## 9.5 Capture Flow for Monetary Items

For any monetary item, the system must use:

Capture → Parse → Preview → Confirm → Commit

The user must see a structured preview before commit unless Auto-Commit mode is active and allowed.

## 9.6 Capture Flow for Non-Monetary Items

For non-monetary items, auto-commit is allowed by default:

Capture → Commit

Examples:

- note
- reminder
- non-valued booking
- attachment
- tag

## 9.7 Confirmation Preview Requirements for Monetary Items

The preview must show:

- entity type
- counterparty
- net amount
- VAT amount
- gross amount
- VAT rate
- category
- relevant date
- any confidence indicator if AI parsing is uncertain

Buttons:

- Confirm
- Edit
- Cancel

## 9.8 Composition vs Committed State

Before confirm, the data is in composition state only.

Composition state:

- is editable
- is not in the Inbox
- does not affect VAT
- does not affect readiness
- does not sync as final financial truth

Once committed:

- entity is created
- event is logged
- VAT/readiness recalculated
- sync triggered
- record becomes subject to immutability rules

# 13. Functional Area 5: Monetary Integrity Rules

## 13.1 Objective

Ensure financial truth remains stable and trusted after commit.

## 13.2 No Deletion

Committed monetary entities cannot be deleted.

## 13.3 No Post-Commit Amount Mutation

Committed monetary amounts cannot be edited in place.

## 13.4 Correction via State Change

# 14. Functional Area 6: Tax Readiness Engine

## 14.1 Objective

Continuously show how ready the current quarter is for export.

## 14.2 Scope

Tax Readiness applies only to the current active quarter.

It does not apply live to:

- past snapshots
- future quarters

## 14.3 Output

Show:

- Tax Readiness percentage
- number of issues remaining
- drill-down into what is missing
- issue severity

Example:

Tax Readiness: 94%  
3 items require attention

## 14.4 Not Gamified

This is a professional operational metric, not a game.

No badges, streaks, confetti, or points.

## 14.5 Drill-Down Requirement

Users must be able to inspect the missing percentage.

Example breakdown:

- missing supplier on 2 receipts
- uncategorised expense
- post-snapshot change requiring new version

Each issue must be actionable and link to the relevant record or flow.

## 14.6 Weighting Principles

Monetary-impacting issues must carry the heaviest weight.

### Tier 1: Critical Monetary Integrity
Examples:

- missing VAT rate
- unconfirmed monetary entry
- incorrect VAT classification
- duplicate monetary item
- post-period change not reflected in a newer snapshot
- unresolved void/correction state impacting totals

### Tier 2: Structural Completeness
Examples:

- missing supplier/client
- missing category
- missing due date
- missing reference number

### Tier 3: Optional Quality Signals
Examples:

- missing attachment
- weak description
- missing notes

## 14.7 Score Behavior

- score updates immediately for current active quarter
- score does not retroactively change stored snapshot scores
- score must be deterministic
- score changes must be explainable

# 15. Functional Area 7: Snapshot & Versioning Engine

## 15.1 Objective

Turn the current quarter into an immutable export package that can be stored, re-downloaded, and versioned if later changes occur.

## 15.2 Snapshot Creation Rules

At least one snapshot must always be possible.

The system should not hard-block export due to unresolved issues.

However, unresolved issues must be attached and surfaced.

## 15.3 Snapshot Contents

Each snapshot must store:

- snapshot_id
- version_number
- quarter_label
- created_at
- created_by
- transaction IDs included
- totals
- VAT totals
- readiness score at time of export
- integrity warning summary
- files generated
- optional checksum/hash if implemented

## 15.4 Snapshot Immutability

A snapshot is frozen once created.

No recalculation.
No silent update.
No overwrite.

If data changes later, that is not a change to the snapshot. It is a change to the live quarter state.

## 15.5 Post-Snapshot Change Logic

If a change occurs in a quarter after snapshot creation:

- flag quarter as changed since last snapshot
- keep previous snapshot intact
- allow user to generate a new snapshot version
- do not auto-generate
- do not overwrite prior version

## 15.6 Versioning Model

Example:

- Q1 2026 Snapshot 001
- Q1 2026 Snapshot 002
- Q1 2026 Snapshot 003

Each version is immutable and separately downloadable.

## 15.7 Diff Engine

When generating Snapshot 002 or later, show changes since last snapshot:

- added transactions
- voided transactions
- adjustments
- revenue impact
- VAT impact

This must be clear before the user confirms generation of the new version.

## 15.8 No Redundant Versions

If nothing changed since the last snapshot, the system should not allow a new snapshot version.

Show:

- No changes since Snapshot 001

## 15.9 Integrity Report

Every snapshot package must include an integrity report.

Example contents:

- all monetary items confirmed
- unresolved warnings
- missing optional fields
- missing counterparties
- readiness score at export

# 16. Functional Area 8: Quarter Lifecycle

## 16.1 Objective

Support clear quarter-state behavior without overcomplicating the workflow.

## 16.2 Active Quarter

The quarter is live.
Capture continues.
Readiness changes live.
Snapshot can be generated.

## 16.3 Snapshot as Symbolic Closure

Generating a snapshot effectively symbolizes the quarter being done, but it does not hard-lock the quarter.

The quarter remains open unless the user manually closes it.

## 16.4 Manual Close

Manual Close is optional.

If selected:

- timestamp the close
- prevent new monetary entries in that quarter
- prevent edits that would impact that quarter
- log the event
- disable further snapshot generation until reopened

## 16.5 Reopen

If user needs to reopen a manually closed quarter:

- require explicit confirmation
- require biometric confirmation
- require a reason
- log the reason and action immutably
- reactivate diff logic and readiness updates

## 16.6 Reopen Example

Quarter reopened  
Reason: Late supplier invoice received

## 21.1 Objective

- biometric confirmation

- visible UI state change

# 28. Definition of Trust

For this product, trust means:

- the data is structured
- the data is complete enough to act on
- financial history does not disappear
- committed amounts do not silently change
- quarter exports can be reproduced
- prior versions remain available
- missing items are visible, not hidden
- the accountant does not need to clean up the output

This is the core KPI philosophy of the product.
