  Plan: Real-Time (RT) Trade Filter Management

  This document outlines the plan to implement a new UI and backend logic for managing and applying filters to real-time
  trades before they are saved to JSON.

  1. Understanding of Requirements

  The goal is to create a UI within the trade_monitor application that allows a user to dynamically assign/de-assign
  specific logging filters to active trading strategies ("Trade Titles"). This provides fine-grained control over which
  trades are recorded to JSON, based on predefined rules.

  The core requirements are:
   1. UI for Allocation: A new page to link "Trade Titles" to "Trade Filters".
   2. Source of Titles: The list of trade titles should be sourced from a master list of "active" strategies.
   3. Source of Filters: The list of available filters will be sourced from the config.json file.
   4. Flexible Allocation: The UI must support assigning filters to individual titles, all titles at once, and
      de-allocating them.
   5. Drill-Down/Analysis: A feature to select a filter and view all the trades that were saved to JSON because they
      passed that filter, along with their performance.
   6. Confirmation: Confirm that the existing trade closure process correctly removes the "open" trade JSON file.

  2. Confirmation of Existing Logic

  > Requirement: "...whenever a trade is sent to json the trade closure process that an opposite json file closes of the
  open json appropriately (this should already be addressed but confirm!!)"

  Confirmation: Yes, this is addressed. The close_current_trade and close_reverse_trade functions in
  simple_trend_trader.py already contain the necessary logic. When a trade is closed, the script saves its final state
  to the closed_trades folder and then explicitly constructs the path to the corresponding file in the open_trades
  folder (including the date-stamped subdirectory) and deletes it using os.remove().

  3. Detailed Implementation Plan

  This plan assumes trade_monitor is a Python application using the tkinter UI framework, consistent with other
  UI-related scripts in the project.

  3.1. Data Model & Configuration

  The current system is not designed for per-title filter allocation. I will introduce a new data model to handle this.

  1. Redefine `active_filters` in `config.json`:
  The active_filters list in C:\Users\edebe\eds\Trades\config.json will be semantically redefined as
  `available_filters`. It will serve as the master list of all possible filters the system knows about.

  Example `config.json`:

   1 {
   2     "available_filters": [
   3         "prev_closed_same_signal_alt_net_negative"
   4     ]
   5 }

  2. Create New Allocation File: `trade_filter_allocations.json`:
  I will create a new file, Trades/trade_filter_allocations.json, to store the many-to-many relationship between Trade
  Titles and the filters assigned to them.

  Example `trade_filter_allocations.json`:

   1 {
   2   "GBP_TP300_...": ["prev_closed_same_signal_alt_net_negative"],
   3   "EUR_TP200_...": [],
   4   "JPY_TP500_...": ["prev_closed_same_signal_alt_net_negative"]
   5 }

  3.2. UI Implementation (trade_monitor application)

  A new top-level menu item "RT Trade Filter" will be added. Clicking it will open a new window with the following
  components:

  Page Layout:
   * Left Pane: A Listbox widget showing all "active" Trade Titles (sourced from config_trade.json).
   * Right Pane: A frame that populates when a Trade Title is selected. This frame will contain a Checkbutton for each
     filter found in the available_filters list from config.json. The checkboxes will reflect the current allocations
     for the selected title.
   * Action Buttons:
       * Save for Selected: Saves the checkbox states for the currently selected Trade Title to
         trade_filter_allocations.json.
       * Apply to All Active: Assigns the current set of checked filters to all Trade Titles in the list.
       * Clear for Selected: Unchecks and saves an empty filter list for the selected Trade Title.

  Drill-Down Feature:
   * A new Frame at the bottom of the page.
   * A Combobox (dropdown) populated with the available_filters.
   * A Treeview widget (for a table-like display).
   * When a user selects a filter from the Combobox, the application will scan all closed_trades JSON files, find trades
     that were saved while that filter was active for their respective Trade Title, and display them in the Treeview
     with key performance data (e.g., P&L $, Entry/Exit times).

  3.3. Backend Logic (simple_trend_trader.py)

  The trading script will be modified to use the new allocation file.

   1. Load Allocations: At startup, the script will load trade_filter_allocations.json into a dictionary.
   2. Modify `save_trade_state_to_json`: The call to trade_filters.apply_filters() will be enhanced. Instead of passing
      the global available_filters list, it will:
       * Get the Trade Title from the trade_data object.
       * Look up the specific list of active filters for that title from the loaded allocations dictionary.
       * Pass this specific list to apply_filters(). If a title has no entry, it will default to an empty list (no
         filters applied).
