# Plan for Re-introducing Console Logging to `sp_001_zone_distribution_trade`

**Version:** 64.2.18  
**Date:** 2025-11-08

## 1. Requirement

The user needs to see real-time console output from the `dbo.sp_001_zone_distribution_trade` stored procedure to monitor its execution flow and debug its behavior. This functionality was desired to be similar to a previous version of the procedure that provided console messages.

## 2. Problem Analysis

Directly adding `PRINT` statements with nested `FORMATMESSAGE` calls in previous attempts resulted in T-SQL syntax errors (`Incorrect syntax near ')'`), making the stored procedure non-deployable. The existing mechanism of inserting logs into a temporary table (`#debug_log`) and selecting from it at the end does not provide the real-time, interleaved console output required for live monitoring.

The core challenge is to output formatted messages to the console at various points during the procedure's execution without causing syntax errors.

## 3. Plan of Approach

The approved solution is to use a "staging variable" for log messages before they are displayed. This approach is robust and avoids the T-SQL parsing issues encountered previously.

1.  **Declare a Staging Variable:** A new `nvarchar(max)` variable, `@log_message_temp`, will be declared at the beginning of the procedure.
2.  **Decouple Formatting from Printing:** At every point where a log message is generated, the process will be broken into three distinct steps:
    a.  The formatted message string will be created using `FORMATMESSAGE` and assigned to the `@log_message_temp` variable.
    b.  The value of `@log_message_temp` will be inserted into the `#debug_log` table for persistent logging within the session.
    c.  The value of `@log_message_temp` will then be sent to the console using a `PRINT` statement.
3.  **Remove Final SELECT:** The `SELECT` statement at the end of the procedure, which was previously used to dump the log table, will be removed. Its function is now superseded by the real-time `PRINT` statements.
4.  **Documentation Tags:** All new or modified lines related to this change will be tagged with a comment indicating the version and date for future reference (e.g., `-- 2025-11-08: 64.2.18`).

This method ensures that the `PRINT` command receives a simple variable, avoiding the complex nested function calls that caused the original syntax errors, while fulfilling the user's requirement for real-time console output.

## 4. List of Changes

*   **File:** `db_scripts/dbo.sp_001_zone_distribution_trade.StoredProcedure.sql`
    *   **Change 1:** Declare the new staging variable `DECLARE @log_message_temp nvarchar(max);` near the top of the procedure.
    *   **Change 2:** Systematically refactor every `INSERT INTO #debug_log ...` statement to use the new three-step process (SET variable, INSERT, PRINT).
    *   **Change 3:** Add versioning comments (`-- 2025-11-08: 64.2.18`) to the new `PRINT` statements.
    *   **Change 4:** Remove the final `SELECT log_time, message FROM #debug_log ORDER BY log_time;` statement from the end of the procedure.

## 5. Post-Implementation Analysis & Findings

After implementing the console logging, the detailed logs for `s_g2` and `b_g2` values were still not appearing. The investigation revealed the following:

1.  **Initial Misdiagnosis:** The problem was initially thought to be a logic error in the stored procedure's product-processing loop.
2.  **Root Cause Discovery:** Through a series of direct database queries, it was confirmed that the issue was not with the code but with the underlying data.
3.  **The Actual Problem:** The `dbo.zone_distribution_snapshots` table in the `tradedb_sim2` database contains **zero rows** for the products being processed (including `eurcad_c`, `gbpnzd_s`, and `gbpcad_c`).
4.  **Conclusion:** The stored procedure is functioning correctly. It identifies that there is no data for the specified products and, as a result, correctly skips the signal generation logic for them. The log message `Not enough zone data (found 0 rows)` is the accurate and expected behavior in this scenario. The core issue lies with the data population process for the `tradedb_sim2` database, which is not providing the necessary snapshot data for the configured products.
