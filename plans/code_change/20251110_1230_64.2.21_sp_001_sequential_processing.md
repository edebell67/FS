# **Plan: `sp_001_zone_distribution_trade` Refactor for Sequential Snapshot Processing**

**Version:** 64.2.21  
**Date:** 2025-11-10  
**Objective:** Refactor `dbo.sp_001_zone_distribution_trade` so every new `vw_201_zone_distribution_snapshots` row is processed in strict chronological sequence. This closes the gap created by the previous `TOP 2` sampling which skipped intermediate snapshots (and therefore missed trade signals) when multiple rows landed between procedure runs.

---

### **Part 1: Prerequisite - Database Schema Change**

*   [x] Create a Tracking Table
    *   **Action:** ship `db_scripts/create_sp_001_tracking_table.sql` and run it once per environment.
    *   **Purpose:** defines `dbo.sp_001_last_processed_snapshot`, a bookmark table tracking the most recent snapshot time seen per product.
    *   **Table Schema:**
        ```sql
        CREATE TABLE [dbo].[sp_001_last_processed_snapshot](
            [product] [nvarchar](50) NOT NULL PRIMARY KEY,
            [last_processed_time] [datetime2](3) NOT NULL
        );
        ```

### **Part 2: Stored Procedure Refactoring (`sp_001_zone_distribution_trade`)

Processing order shifts from `G-Level -> Product -> Snapshots` to `Product -> Snapshots -> G-Level`, ensuring we compare every chronological pair before emitting signals.

*   [x] 1. Modify Product Loop
    *   **[x] Pre-materialize View:** At procedure start copy `vw_201_zone_distribution_snapshots` into `#vw_snapshot` with an explicit schema to avoid repeated view hits mid-loop.
    *   **[x] Get Bookmark:** For each product read `last_processed_time` from `dbo.sp_001_last_processed_snapshot`. When no record exists seed the bookmark with `GETDATE()` so we only react to snapshots generated after the feature goes live (no historical replay by default).
    *   **[x] Build Snapshot Batch:** Create `#snapshot_batch` with a descending `row_num` (via `ROW_NUMBER() OVER (ORDER BY snapshot_time DESC)`) and copy **all** snapshots whose `snapshot_time > last_processed_time`. Each row carries the full set of `s_G*`, `b_G*`, and `latest_price` fields so downstream comparisons never reference missing columns.
    *   **[x] Handle Empty Batch:** If the batch has fewer than two rows, log-and-skip without raising errors so the bookmark stays unchanged until a complete pair arrives.

*   [x] 2. Implement Snapshot-Comparison Loop
    *   Iterate `row_num` in descending order (newest first) and feed each adjacent pair through the existing G-level loop.
    *   The BUY/SELL tests continue to look for the "number -> NULL" transition (e.g., `previous_s_g > 0` while `current_s_g IS NULL`) which, given the descending ordering, reflects “latest numeric immediately followed by older NULL”.

*   [x] 3. Update the Bookmark
    *   After finishing the batch for a product, persist the newest processed timestamp (`MAX(snapshot_time)` from the batch) via `MERGE` into `dbo.sp_001_last_processed_snapshot`.

*   [x] 4. Cleanup
    *   Legacy `#zone_data` temp-table usage remains removed; sequential processing now relies exclusively on `#vw_snapshot` and `#snapshot_batch`.

---

### **Notes & Assumptions**

1. Bookmark defaults to `GETDATE()` so we intentionally ignore pre-existing snapshot history when the refactor first ships. Seed the table manually with an older timestamp if a historical replay is ever required.
2. `vw_201_zone_distribution_snapshots` stays the authoritative source; we rely on the view’s continuous updates and do not bypass it.
3. Snapshot ordering is purely time-based; no surrogate key tie-breakers exist today. If two rows share the same `snapshot_time`, their relative order inside `ROW_NUMBER()` is non-deterministic, but the "number followed by NULL" requirement is satisfied so long as times advance monotonically.

### **Debugging Status**

* The previous `SELECT ... INTO #snapshot_batch` returned zero rows because we constrained the query to `TOP 2` and limited the columns to `s_G9/b_G9`, which clashed with downstream logic. Moving to the full-schema materialized temp table plus `ROW_NUMBER()` resolves the empty batch symptom.
* Remaining validation involves executing the stored procedure end-to-end in the target environment to confirm the bookmark advances, `#snapshot_batch` fills as expected, and signals fire when the number->NULL transition occurs.
