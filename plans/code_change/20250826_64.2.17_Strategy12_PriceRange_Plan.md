# API Consistency Plan: Centralized DB Suffix Application

**Date:** 2025-08-26
**Version:** 64.2.20

## 1. Problem Description

Previously, the application suffered from inconsistent application of the `db_suffix` (e.g., `?db=tradedb_sim` or `?db=tradedb`) to API endpoints. This led to several issues:
*   **Scattered Logic:** The logic for appending the `db_suffix` was spread across various `fetch` methods in `tradepanel/api.py`.
*   **Inconsistency:** Some methods manually appended the suffix, some relied on the constant definition, and some completely omitted it.
*   **Bugs:** This inconsistency resulted in critical bugs, such as fetching live data (forex prices, open/closed trades) when the application was running in simulation mode, leading to inaccurate simulation results.

## 2. Solution: Centralized `_get_final_api_url` Helper

To permanently resolve this, a new centralized helper method, `_get_final_api_url`, has been implemented within the `APIClient` class in `tradepanel/api.py`. This helper is now responsible for dynamically appending the correct `db_suffix` to API endpoints based on the `APIClient`'s `is_simulation_mode`.

### Key Principles of the Solution:
*   **Centralization:** The logic for determining and appending the `db_suffix` is now in one place.
*   **Consistency:** All relevant API fetching and posting methods now consistently use this helper.
*   **Robustness:** Reduces the likelihood of future `db_suffix` related bugs due to oversight.
*   **Non-Interference:** Specific cases where the `db_suffix` is already handled (e.g., `ENDPOINT_FOREX_PRICES_APP_SIM` or methods that manually append it for specific sorting/filtering) are carefully managed to avoid double-suffixing.
