This document outlines the plan to implement thread-safe UI updates to prevent access violations, as requested on 2025-08-29.

## 1. Understanding of Requirements

The user reported a `Windows fatal exception: access violation` crash in the `tradepanel` application. Analysis of the log indicated that the crash occurred during UI updates, specifically when `tksheet` was being manipulated, and was likely caused by UI updates being performed from a background thread (the `TradeManager`'s data processing loop) directly on Tkinter widgets, which are not thread-safe.

## 2. Plan of Approach

The core of the solution is to ensure all UI updates originating from background threads are marshaled back to the main Tkinter thread. This will be achieved by introducing a generic UI update queuing mechanism in `main_window.py` and modifying `trade_engine.py` to use this mechanism for all its UI interactions.

1.  **Introduce a generic `queue_ui_update` method in `main_window.py`**: This method will use `self.root.after(0, func)` to schedule any given function to run on the main Tkinter thread at the earliest opportunity.
2.  **Modify `trade_engine.py` to use `queue_ui_update`**: All calls to `self.ui_proxy.update_trade_display()` and `self.ui_proxy.update_product_trade_panel()` within `TradeManager` methods (`execute_close_trade`, `execute_automated_buy_trade`, `execute_automated_sell_trade`) will be replaced with calls to the new `queue_ui_update` method, passing the original UI update function as a lambda.
3.  **Update Version Number**: Increment the version number in `Constants.py`.

        def queue_ui_update(self, func: Callable[[], None]) -> None:

            2025-08-29, V64.2.30: Safely queues a UI update function to be executed on the main Tkinter thread.
