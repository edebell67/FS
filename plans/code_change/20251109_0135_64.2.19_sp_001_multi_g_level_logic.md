# Plan: `sp_001_zone_distribution_trade` Multi-G-Level Enhancement

**Version:** 64.2.19
**Date:** 2025-11-09

## 1. Understanding of Requirements

The goal is to enhance `sp_001_zone_distribution_trade` to process multiple G-levels for signal generation. The configuration entries `sp_001_zone_buy_var` and `sp_001_zone_sell_var` in `dbo.config` will now be treated as comma-separated lists (e.g., 'g2,g9').

The stored procedure must iterate through each G-level from the configuration. For each G-level, it will execute the complete signal generation logic for all tradeable products.

If a trade is generated, the `trade_reason` column must be dynamically set to reflect the specific G-level that triggered the trade, following the format `sp_001_zone_{g_level}` (e.g., `sp_001_zone_g2`).

## 2. Plan of Approach

The modification will involve wrapping the existing product-processing logic within a new, outer loop that iterates through the specified G-levels.

1.  **Parse G-Levels:** At the start of the procedure, I will read the `sp_001_zone_buy_var` and `sp_001_zone_sell_var` config values. I will then parse these comma-separated strings and store the distinct G-levels into a new temporary table, `#GLevelsToProcess`.
2.  **Outer G-Level Loop:** I will introduce a new `CURSOR`-based `WHILE` loop that iterates through each G-level in the `#GLevelsToProcess` table.
3.  **Nest Existing Logic:** The existing logic that loops through each tradeable product (the `@product_cursor` loop) will be moved entirely inside this new G-level loop.
4.  **Dynamic SQL and `trade_reason`:**
    *   Inside the loop, the dynamic SQL that constructs the queries for fetching `s_g_val` and `b_g_val` will be modified to use the current G-level from the loop variable.
    *   The `INSERT` statement that creates new trade candidates in the `#cand` table will be updated to dynamically set the `trade_reason` by concatenating `'sp_001_zone_'` with the current G-level (e.g., `sp_001_zone_g2`).
5.  **Cleanup and Refactoring:** The single-value `@buy_g_level` and `@sell_g_level` variables will be removed, as their function is replaced by the new loop variable. I will ensure all temporary tables are correctly created and cleared within the nested loop structure to prevent data from one iteration affecting the next.

This approach will process each G-level independently against all products, ensuring the logic is clean and meets the requirements.
